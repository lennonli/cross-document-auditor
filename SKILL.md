# 跨机构底稿智能交叉审计系统 (Cross-Document Auditor v3.0)

## 🎯 核心使用场景与触发条件
当用户指令涉及以下意图时，必须强制触发本技能工作流：
- “对比招股说明书与法律意见书/审计报告的差异”
- “核对多份底稿之间的数据打架或事实矛盾”
- “审查底稿是否真实完整地回答了监管的问询要求”
- “看 A 文件有没有覆盖 B 文件的内容”

## ⚠️ 核心纪律（防呆原则）
本技能是对无数次项目实战血泪史的终极凝练，执行时**绝对不可绕过**以下原则：
1. **绝对禁止全文逐字对比**：严禁建议用户使用 Beyond Compare 等差异比对工具。券商、律所行文习惯不同，全文比对会产生95%的无效噪音。
2. **强制双向集合核查（Bidirectional Check）**：凡遇清单类数据（关联方、注销企业等）必须计算“A独有”和“B独有”的差集，单向检索必定导致漏报！
3. **禁止复杂表格扁平化搜索**：财务数据不能转为纯文本或普通 Markdown 搜索，必须提取为带有坐标的二维结构。
4. **强制拦截重大性遗漏（Materiality）**：对“A有、B无”的信息不能机械忽略，必须引入大模型常识判定其是否属于重大信披缺失（如资金占用、重大违规罚款）。

## 🛠️ 标准操作程序 (SOP - 10步验稿法)

### 阶段一：基础底稿解析与内部排雷 (Extraction & Self-Consistency)
**第一步：深度文档结构化提取 (Deep Extraction)**
不可仅提取最终净稿，必须同时捕捉隐藏信息：
- **纯净版提取**：转换 docx 为规范 markdown。
- **底层修订痕迹提取**：使用 `pandoc --track-changes=all` 专门抓取被“划掉/删除”的实体名称，寻找被中介机构偷偷删掉的“靶标”。
- **表格二维提取**：使用脚本剥离财务与股权数据表为 DataFrame 数组，锁定 `[实体名称, 年份, 科目]` 三维坐标，防止行列错位。

**第二步：单文件内部自洽性扫描 (Intra-Document Consistency)**
跨文件对比前，必须先“除内鬼”：
- **实体定性自洽**：如同一个文件前文写“当前控制”，后文写“已注销”，立即作为 P0 级矛盾报警。
- **数字全局自洽**：如同文件内同一指标（如在册员工人数 245 vs 252）出现前后不一致，立即标记。

### 阶段二：静态交叉比对 (Static Cross-Document Diffing)
**第三步：实体清单的“双向差集”核对 (Entity Set Math)**
针对所有列举型事实（董监高名单、关联方名单、对外投资、专利明细）：
- 提取 A 文件全集 Set_A，提取 B 文件全集 Set_B。
- 输出 `Set_A - Set_B`：找出被 B 文件遗漏核查的实体。
- 输出 `Set_B - Set_A`：找出被 A 文件遗漏核查的实体。（防范“券商披露20家，律师只查18家”的事故）。

**第四步：结构化属性精准对撞 (Attribute Collision)**
建立“全局别名词典”（例：`发那特 = 抚州发那特信息技术服务有限公司`）：
- **比例卡位**：对特定主体的持股/控制比例进行硬核对，拦截 `100% vs 80%` 这类致命认定差异。
- **金额卡位**：按 `[交易对手, 交易年度]` 锁定金额，拦截 `500.00 vs 3,500.00` 的统计笔误。

### 阶段三：动态逻辑深度验证 (Advanced Dynamic Validation)
**第五步：跨文件“财务勾稽关系”验算 (Cross-Document Articulation)**
不只做相等判定（==），强制引入代数求和判定（SUM）：
- 当 A 文件呈现聚合总数，B 文件呈现离散明细时，系统自动加总 B 文件的明细数据（如：子公司1+子公司2采购额），再与 A 文件的“合并报表采购总额”核对闭环。

**第六步：全量事件“时间线向量断层检测” (Chronological Vectoring)**
针对历史沿革、股权流转等高危章节：
- 对目标公司抽取带日期（年月日）的事件，生成时间轴向量序列。
- 对比 A、B 向量。如发现 `[2010代持 -> 2017解除]` vs `[2010代持 -> 2013增资 -> 2017解除]`，立即爆出“历史发展存在时间断层”的严厉预警。

**第七步：责任主体口径降噪过滤 (Verification Boundary Filter)**
提取比对数据的主语与核查视角：
- 若数据主体分别是“保荐机构核查覆盖率 75%”与“本所律师核查覆盖率 68%”，系统自动识别由于核查手段口径不同导致的差异，将其从 P0 级矛盾降级为“合理差异”，避免无效报警噪音。

### 阶段四：终极溯源与防篡改 (Ultimate Grounding & Anti-Tampering)
**第八步：外部开源数据穿透验证 (External Truth Grounding)**
- 打破“底稿信息孤岛”。提取出注册资本、成立时间、诉讼情况等核心事实后，强制调用网络搜索或企查查/天眼查能力，将 `A文件数据 == B文件数据` 升维到 `A == B == 外部物理世界真实数据`。

**第九步：非结构化承诺“向量语义防篡改” (Vector Embedding Anti-Tampering)**
针对同业竞争承诺、兜底豁免条款等大段非结构化文本：
- 将双方段落转化为高维空间向量计算余弦相似度。相似度低于阈值，立刻查明是否有人偷偷增加了“原则上”、“除另有约定外”等卸责定语。

**第十步：监管问询函的逆向闭环工程 (Inquiry Letter Reverse Engineering)**
引入前置触发条件 Document C（《审核问询函》）：
- 提取监管问询的全部逻辑拆解点（如：必要性、公允性、利益输送）。
- 逆向扫描 A、B 底稿，验证是否每一小问都得到了正面、实质性回应。发现避重就轻者，立即预警。

## 📊 输出报告分级规范
给用户交付的比对报告必须极其克制且严密分级：
1. **P0 级 - 实质性矛盾与内伤**：（例：持股比例打架、同一文件自相矛盾、清单缺漏、时间线断层）。要求：立即修改底稿。
2. **P1 级 - 重大信披规避与遗漏**：（例：监管问询避重就轻、大额资金占用未批、承诺条款遭篡改）。要求：强制增补或重写。
3. **P2 级 - 口径与详略差异**：（例：中介覆盖率口径不同、第三方对比价格细节缺失）。要求：中介备齐工作底稿备查即可。
4. **验证通过项**：必须列示“已严格核对、勾稽关系闭环且外部溯源一致”的核心财务与合规结论，给予合伙人签字的安全感。
